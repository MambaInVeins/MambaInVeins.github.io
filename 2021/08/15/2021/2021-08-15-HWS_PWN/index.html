堆栈利用之对抗linux保护

linux常见保护机制

Canary：函数开始执行的时候往栈里插入Canary值，函数返回时验证该值是否合法，防止栈溢出
Fortify：防止格式化字符串漏洞，包含%n的格式化字符串不能位于程序内存中的可写地址。当使用位置参数时，必须使用范围内所有参数。
NX/DEP：堆栈不可执行
PIE/ASLR：地址空间分布随机化，增加ROP利用难度
RELRO：设置符号重定向表为只读，并且在程序启动时就解析并绑定所有动态符号，从而减少对GOT（Golbal Offset Table）的攻击。但是还是可读的，可以进行信息泄露。

栈

基础栈结构

  • 后进先出 push/pop

  • 用于保存函数调用信息和局部变量

  • 由高地址往低地址写入

传参

  • x86:函数参数保存在栈上，在函数返回地址上方
  • x64:优先寄存器传参，前六个参数rdi rsi rdx rcx r8 r9，更多参数才会保存在栈上

函数调用

进入函数时

call xxx
    push retaddr(call的下一条指令地址，要知道怎么回来)
    jump xxx

xxx
    push rbp（保存栈，好恢复）
    mov rbp,rsp（栈底抬上去）
    sub rsp,xxh（抬高栈顶给局部函数预留栈空间）

退出函数时

leave
    mov rsp,rbp（把rsp弄回来）
    pop rbp（把rbp弄回来）

ret
    pop rip（这个时候rip就被retaddr的值取代了）

栈溢出

程序向栈中某个变量中写入的字节数，超过了这个变量本身申请的字节数，导致与其相邻栈上的变量被改变。轻则崩溃，重则控制执行流程。

前提：程序向栈上写入数据；写入数据的大小没有被控制。

栈的字节对齐

栈的字节对齐，实际是指栈顶指针必须须是16字节的整数倍。栈对齐帮助在尽可能少的内存访问周期内读取数据，不对齐堆栈指针可能导致严重的性能下降。

上文我们说，即使数据没有对齐，我们的程序也是可以执行的，只是效率有点低而已，但是某些型号的Intel和AMD处理器对于有些实现多媒体操作的SSE指令，如果数据没有对齐的话，就无法正确执行。这些指令对16字节内存进行操作，在SSE单元和内存之间传送数据的指令要求内存地址必须是16的倍数。

因此，任何针对x86_64处理器的编译器和运行时系统都必须保证分配用来保存可能会被SSE寄存器读或写的数据结构的内存，都必须是16字节对齐的，这就形成了一种标准：

任何内存分配函数（alloca, malloc, calloc或realloc）生成的块起始地址都必须是16的倍数。
大多数函数的栈帧的边界都必须是16直接的倍数。

如上，在运行时栈中，不仅传递的参数和局部变量要满足字节对齐，我们的栈指针（%rsp）也必须是16的倍数。

ret2_libc

ROP（Return Oriented Programming）

有了NX保护之后，堆、栈、bss段没有执行权限…

在栈溢出的基础上，利用程序中已有的小片段gadegts来改变某些寄存器或者变量的值，从而控制程序的执行流程。所谓gadgets就是以ret结尾的指令序列，通过这些指令序列，我们可以修改某些地址的内容，方便连续地控制程序的执行流程。

需要跳转的目标函数如果程序中没有呢，如system，Libc中应有尽有，泄露libc基址

然而ALSR/PIE保护技术使得程序基地址每次加载不一样，如果开启了PIE，连基本的gadgets的地址都不知道，怎么办？结合其他漏洞或者功能进行泄露

ROP(ret2libc)

假设需要利用ROP调用system(‘/bin/sh’)

思路：
1.泄露GOT表中某个函数的libc地址
2.在libc中找到system,'/bin/sh'和这个函数的相对偏移
3.得到system和/bin/sh的地址
4.构造rop链成功利用
注意：
只能泄露已经执行过一次函数的libc地址，因为动态链接库的加载机制是lazy原则,即用时加载，第二次就不用加载了。
在执行了一次某函数之后，GOT表中就会把一个函数在程序中的终极偏移存起来
终极偏移=libc基址（每次加载都不一样）+库内函数相对偏移
system =  libc基址 + system在库中相对偏移
/bin/sh = libc基址 + /bin/sh在库中相对偏移

例子：ret2_libc:pwn3

对抗Canary保护

泄露Canary的值（任意读的功能）

mov rax fs:28h
mov [rsp+38h+var_30],rax
...
mov rcx,[rsp+38h+var_30]

覆写副本值

需要进行fs:28h位置的爆破， 在mapped段，相对于Libc基地址在每次电脑重启时的偏移是一定的，就可以读出来或者覆写掉

劫持stack_chk_fail函数

修改stack_chk_fail的GOT地址，便可以在触发canary检查失败时，跳转到指定位置

stack smashing

当canary被覆盖之后，会call到__stack_chk_fail打印argv[0]这个指针指向的字符串，默认是程序的名字。
如果我们把它覆盖为其他的地址时，它就会把其他内存地址的信息给打印出来。（比如校验时内存里的账号密码等）

这里利用__stack_chk_fail实现任意地址读。
gdb调试下进入___stack_chk_fail函数，再步入__GI__fortify_fail函数，可以看到这里执行了一个叫做__libc_message的函数
功能大概就是输出一些错误信息
其中当前程序的路径名是从argv[0]中取得的并存放在栈上，然后用类似于格式化字符串的方式进行拼接。
只要输入足够长，是可以覆盖__libc_message的参数的，从而把我们覆盖的地址位置的数据打印出来。

BROP逐字节爆破(没有源码及二进制文件)

攻击条件：
（1）远程程序必须先存在一个已知的栈溢出漏洞，并且知道如何触发漏洞
（2）服务器进程在crash之后会重新复活，并且复活的进程不会被re-rand，意思是虽然有ASLR保护，但是复活的进程和之前的进程的地址随机化与canary是一样的。当前像nginx、Mysql、Apache、OpenSSL、Samba等服务器应用都符合这种特性。

基本思路：
1.判断栈溢出的长度
2.逐字节爆破Canary（如果没有开，就跳过这一步）
3.寻找stop gadget
    如果把返回地址覆盖成某些随意选取的内存地址，程序很大可能会crash。但如果返回地址指向了一块代码区域，当程序的执行流跳到那段区域之后，程序不会crash而是进入了无限循环，这时程序仅仅是hang在了那里，攻击者能够一直保持连接状态。这种类型的gadget称为stop gadget。
    可以不断爆破返回地址尝试出stop gadget
    为什么要找stop gadget，因为有了stop gadget，就可以在需要尝试的return address之后填上足够多的stop gadget，那么任何会造成进程崩溃的gadget还是会造成进程崩溃，但是useful gadget就会进入block状态。
4.寻找useful gadget
    useful gadget就是一类可以操控有用的寄存器的gadget。特殊的useful gadget也就是BROP gadgets，需要从栈上pop 6个值然后执行ret，利用之前的stop gadget很容易找到BROP gadget。
    在返回地址之后的六个使程序崩溃的值，再填入stop gadget。很少有连续pop六次。brop gadget偏移9就是 pop rdi;ret;
        pop rbx;
        pop rbp;
        pop r12;
        pop r13;
        pop r14;
        pop r15;
        ret;
5.寻找可用的PLT表项
    如果攻击者发现好多条连续的16字节对齐的地址都不会造成进程crash，而且这些地址+6的地址也不会crash，那么很有可能这就是某个PLT对应的项。
    payload = 'A'*length+canary+p64(pop_rdi_ret)+p64(0x400000)+p64(addr)+p64(stop gadget)
    在程序还没有开启PIE保护的情况下，0x400000处为ELF文件的头部，内容是\x7fELF，可以判断addr是不是puts@plt
6.利用PLT表中的puts（或者write）函数，配合useful gadget，来远程dump信息
    不断循环输出程序内存信息 p64(0x400000)+p64(addr) -> p64(target)+p64(puts)
    payload = 'A'*length+canary+p64(pop_rdi_ret)+p64(target)+p64(puts)+p64(stop gadget)

例子：BROP:pwn4

leakbuff_length 溢出长度为72
Get_Stop_Gadgets 获取stop gadgets, 有些地址返回到main，有些正常结束，有些悬挂住，这些都可以。
Get_Brop_Gadget 获取useful gadgets，比较好找的时brop gadgets，+9的位置就是pop_rdi_ret,不过需要验证一下
Get_Put_addr 获取puts_plt
leak泄露某地址内存，dump_memory不断泄露整块地址，可以dump整个程序
把 dump 下来的文件用 IDA 的二进制模式打开，然后把基址改成 0x400000（编辑->段->重新设置基址），在 puts_plt 处，摁下 c 识别成汇编格式，可以看到 jmp 的地址 puts_got, 知道了这个地址，再用 libcsearch 去进行 ret2libc 就可以了

一些技术

栈劫持（栈迁移）

溢出长度不够，或者想在bss段、堆段等已知位置进行写入，就可以提前进行栈布局。栈劫持其实就是修改了rsp指针。

方法：通过覆盖栈上存储的saved rbp和saved rip，就可以将栈进行劫持。

leave
    mov rsp,rbp (把rsp弄回来)
    pop rbp (把rbp弄回来)
ret
    pop rip

例子：stack migration:migration

解题思路：
1.通过劫持ebp和esp将栈劫持到bss段
2.利用puts函数泄露libc内存空间信息，得到system函数在内存中的地址 ,顺便将栈劫持到另一个地方
3.通过read函数读入"/bin/sh"字符串 然后返回调用system函数getshell

ret2_csu

原理：

__libc_csu_init是一般的程序都会调用libc中的函数，而此函数便是来将libc初始化的，几乎是每一个程序所必备的。我们在构造ROP链的时候，往往会用到能够给寄存器赋值的gadget，形如pop|ret。
__libc_csu_init函数内也存在这样的gadget，我们称之为通用gadget，于是一种利用该通用gadget来构造ROP的方法便诞生了，名为ret2csu。

gadget1:
    pop     rbx
    pop     rbp
    pop     r12
    pop     r13
    pop     r14
    pop     r15
    retn
gadget2:
    mov     rdx, r13
    mov     rsi, r14
    mov     edi, r15d
    call    qword ptr [r12+rbx*8]

gadget1 我们可以利用栈溢出构造栈上数据来控制 rbx,rbp,r12,r13,r14,r15 寄存器的数据。

https://ctf-wiki.org/pwn/linux/user-mode/stackoverflow/x86/medium-rop/#ret2csu

基本利用思路如下

利用栈溢出执行 libc_csu_gadgets 获取 write 函数地址，并使得程序重新执行 main 函数
根据 libcsearcher 获取对应 libc 版本以及 execve 函数地址
再次利用栈溢出执行 libc_csu_gadgets 向 bss 段写入 execve 地址以及 '/bin/sh’ 地址，并使得程序重新执行 main 函数。
再次利用栈溢出执行 libc_csu_gadgets 执行 execve('/bin/sh') 获取 shell。

例子：ret2_csu:ret2csu

ret2reg

原理

查看溢出函返回时哪个寄存值指向溢出缓冲区空间
然后反编译二进制，查找 call reg 或者 jmp reg 指令，将 EIP 设置为该指令地址
reg 所指向的空间上注入 Shellcode (需要确保该空间是可以执行的，但通常都是栈上的)

攻击简介：
ret2shellcode需要事先确定shellcode的地址作为返回地址，若开了ASLR，则难以确定别的机器上的栈（esp）地址。ret2reg则可以绕过ASLR，通过jmp 寄存器的方式跳转到shellcode。

例子：ret2_reg:ret2reg

示例程序中，main函数调用子函数，子函数将输入拷贝到buffer中去，导致缓冲区溢出覆盖子函数返回地址，刚好此时eax指向shellcode，则通过call eax跳转到shellcode。

echo 2 > /proc/sys/kernel/randomize_va_space命令打开了地址随机化，但仅对栈空间，堆地址和动态库加载空间进行随机化，没有对程序做地址随机化。Linux gcc编译器提供了-fPIE选项，可使程序空间做地址混淆，但一般的开源软件和商用Linux发行商的服务进程并没有使用-fPIE进行安全增强。

JOP/COP

原理

换汤不换药，把使用的的代码片段从ret结尾拓展到jmp/call结尾
JOP（Jump Oriented Programming）
    pop esi ; jmp dword [esi-0x70]
COP（Call Oriented Programming）
    mov edx,dowrd [esp+0x48] ; call doword [eax+0x10]

SROP

原理：

Srop 的全称是Sigreturn Oriented Programming
Srop 可以理解成一种高级的ROP，利用了linux下15号系统调用的->rt_sigreturn
Signal是Unix系统中的一种通信机制，通常用于在进程之间传递信息，也可以说是软中断信息
常见于在一个进程中，内核向其发送发送软中断信号，该进程将暂时被挂起，系统进入内核态
因为是暂时被挂起，所以系统会保留该进程的上下文，将所有的寄存器压入栈中，以及signal信息和指向sigreturn的系统调用地址在栈顶上放置rt_sigreturn

漏洞利用点

Signal Frame 被放置在用户进程的内存空间中，也就说Signal Frame是可以读写的
在恢复Signal信号的时候没有检测，也就是说我们可以通过改变Signal Frame中的信息来劫持控制流

例子：SROP:pwn5
